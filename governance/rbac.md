# 角色權限模型(RBAC)

> **三層權限分離**:內部編輯層 → 外部審閱層 → 公開讀取層
> **適用範圍**:Notion 後台、Super.so 公開頁、Tally 表單後台、SQLite 本地備份

---

## 一、三層架構

```
┌─────────────────────────────────────────────┐
│  公開讀取層(Super.so + GitHub Pages)        │
│  匿名訪客可瀏覽:議題頁、條文頁、PDF 下載     │
└─────────────────────────────────────────────┘
              ↑(發布管道)
┌─────────────────────────────────────────────┐
│  外部審閱層(Notion 訪客,最多 100 位)       │
│  專家、合作 NGO、青少年顧問小組可評論         │
└─────────────────────────────────────────────┘
              ↑(內容管道)
┌─────────────────────────────────────────────┐
│  內部編輯層(Notion Plus,2 名編輯位)         │
│  主編 + 研究助理擁有寫入權                   │
└─────────────────────────────────────────────┘
```

## 二、RBAC 詳細矩陣

| 角色 | 議題草稿 | 議題發布 | 兒少敏感資料 | 影子報告匯出 | 刪除權 |
|---|:---:|:---:|:---:|:---:|:---:|
| 主編 | ✅ | ✅ | 去識別化 ✅ | ✅ | DPO 授權後 |
| 研究助理 | ✅ | 提交審核 | 去識別化 ✅ | 草稿 ✅ | ❌ |
| 外部專家審閱 | 評論 ✅ | ❌ | ❌ | ❌ | ❌ |
| 兒少顧問小組 | 兒少版評論 ✅ | ❌ | ❌ | ❌ | ❌ |
| 資料保護長 DPO | ✅ | ❌ | ✅(稽核) | ✅ | ✅ |
| 兒少保護專員 | ❌ | ❌ | 安全旗標 ✅ | ❌ | ❌ |
| 一般訪客 | ❌ | 公開頁讀取 | ❌ | PDF 下載 | 自己資料 ✅ |

## 三、特別規則

1. **「兒少敏感資料」雙人原則**:任何讀取需 DPO + 兒少保護專員兩人同時在場(視訊或實體),並登入稽核 log。
2. **AI 工具預設只能讀 `submission_redacted`**:`submission_raw` 對 AI 完全封鎖。
3. **公開發布雙人簽核**:主編草稿 → 專案經理覆核 → DPO 隱私檢查 → 上線。
4. **撤銷權限**:成員離職、外部專家任期屆滿、青少年顧問離團 → 24 小時內撤銷所有平台權限。
5. **稽核日誌**:Notion / Super.so / Tally 後台啟用稽核紀錄;每月 DPO 抽查。

## 四、Notion 實作步驟

1. 工作區層級:邀請主編、研究助理為 `Workspace Owner`、`Member`。
2. 對 13 張資料庫設定 `Locked`,只有主編能解鎖修改 schema。
3. 對「兒少敏感」資料庫(submission_raw、case_story 中的極敏感標籤)使用 Notion 的 `Restricted` 屬性,只 share 給 DPO + 兒少保護專員。
4. 外部專家透過 `Guest` 邀請,僅 share 特定頁面而非整個工作區。
5. 公開頁透過 Super.so 連動 — Super.so 只能讀 Notion 已標記 `Published = TRUE` 的頁面。

## 五、緊急情境

| 情境 | 處置 |
|---|---|
| 帳號被盜 | 立即撤銷該帳號權限 + 改密碼 + 24h 通報 DPO |
| 敏感資料誤發布 | 立即下架 + 1h 內向當事人通知 + 24h 內向 DPO 報告 |
| 主編離職 | 交接會議 + 7 日內撤銷權限 + DPO 確認資料無外流 |
| 司法調閱要求 | 律師審查必要性 + 律師回覆,**不**主動配合越權索取 |
